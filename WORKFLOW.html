6. Supporting Registers

/workers, /incidents, /improvements — Operational evidence that feeds compliance

Workers

3 workers. Track screening expiry, training records, supervision logs. Feeds: WF-001 (screening), WF-004 (orientation), WF-006 (training).

Incidents & Complaints

Log incidents, investigate, resolve, link to improvements. Feeds: GOV-005 (incident register), GOV-010 (complaints register).

Continuous Improvement

Record improvements from incidents/complaints/feedback. Shows corrective actions + evidence. Feeds: GOV-003, GOV-004.

These registers satisfy checklist items "in_tool" — the auditor can view them directly in the Audit Pack portal without separate document uploads.

10. System Architecture

Technical stack & data flow

LayerTechnologyDetails
FrontendNext.js 16 (App Router)Tailwind 4 + shadcn/ui components
BackendNext.js API RoutesServer-side, same deployment
DatabasePostgreSQL (local → RDS)Prisma ORM. 15 standards, 52 rules, 59 KB docs, 28 checklist items
AIVertex AI (Gemini 2.5 Flash)Via proxy on EC2. Classification, generation, gap analysis
File StorageLocal (→ S3)public/uploads/ for now. S3 ap-southeast-2 for production
Knowledge Base59 NDIS documents indexedLegislation, operations, compliance guides. AI cites by Doc ID
Hosting (planned)AWS EC2 ap-southeast-2Same instance as 24seven backend. Australian data sovereignty

Data flow on upload:

User drops file → POST /api/evidence (saves file + DB record) → returns immediately
  ↓ background (3-5s)
classifyAndLink() → AI reads title+category → maps to standards → upserts EvidenceStandardLink
  ↓
evaluateChecklist() → AI determines which 28 checklist items this doc satisfies → upserts ChecklistStatus
  ↓
evaluateAllRules() → rule engine checks all 52 rules against current evidence state → updates isPassing
  ↓
Next page load shows updated: checklist ticks, standards status, rules score
